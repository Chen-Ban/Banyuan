# ADR-019: 跨平台路径——JSON 中间表示 + 统一接口层 + 各平台适配

> 状态：已接受
> 日期：2025-05-21
> 决策者：chenxin176

## 背景

Banyan 是一个零代码 + AI 设计平台，用户通过拖拽和自然语言操作，产出两份纯数据描述：

- **UI JSON（pages）**：视图树、样式、布局、数据绑定
- **Flow JSON（schema）**：逻辑节点、连线、条件分支、执行参数

80% 的内容由 AI 生成，20% 由用户微调。人不写代码——这意味着我们不需要设计一门编程语言，也不需要通用 AST 到目标语言的编译器。

问题：当 Banyan 要支持构建跨平台应用（Web、macOS、Windows、iOS、Android）时，跨平台渲染和逻辑执行应该以什么方式实现？

## 决策

采用「JSON 中间表示 + 统一接口层 + 各平台适配实现」的路径。不自建语言，不走编译到原生代码的路线。

## 核心认知

**UI JSON 和 Flow JSON 本身就是平台无关的中间表示。** 它们描述的是「画什么」和「做什么」，不涉及「怎么画」和「怎么执行」。这两份 JSON 的地位等同于 Flutter 的 Display List——是一套有限的、声明式的指令集，而非图灵完备的通用代码。

因此跨平台的实现路径被大幅简化：不需要「源代码 → AST → 目标平台代码」的编译流水线，只需要「JSON → 统一接口调用 → 平台原生渲染/执行」的解释执行路径。

## 为什么不自建语言

**1. 用户不写代码。** 自建语言的前提是有人类开发者需要阅读和编写它。Banyan 的用户是设计师和业务人员，他们操作的是可视化界面和自然语言，产出的是 JSON 数据。没有人需要一门新语言。

**2. JSON 已经是足够好的中间表示。** UI JSON 描述的是有限的视觉原语（矩形、文本、图片、容器、滚动列表……），Flow JSON 描述的是有限的逻辑原语（条件分支、循环、HTTP 请求、数据映射……）。这些指令集是封闭的、可枚举的，不需要图灵完备的表达能力。

**3. 解释执行比编译更适合这个场景。** 编译到原生代码适合性能敏感 + 长期运行的场景。而 Banyan 构建的应用本质上是「数据驱动的声明式 UI + 事件触发的有限逻辑流」，解释执行的开销完全可接受，且带来热更新、动态加载等灵活性。

## 统一接口层的职责

统一接口层是 banvasgl 和 flow 引擎需要定义的核心契约。它不关心底层是什么平台，只定义「上层 JSON 需要什么能力才能被正确渲染/执行」。

渲染侧需要的原语（banvasgl Renderer 抽象）：

- 创建/销毁视图节点
- 布局计算（盒模型、flex、绝对定位）
- 图形绘制（矩形、圆角、路径、文本、图片）
- 事件绑定（点击、滑动、输入）
- 数据绑定（视图属性与数据源的响应式连接）
- 动画驱动（属性过渡、关键帧）

执行侧需要的原语（flow FlowRunner 抽象）：

- 节点求值（计算节点输出）
- 条件分支（布尔表达式求值）
- 数据映射（JSON path 取值/赋值）
- 副作用执行（HTTP 请求、本地存储、导航跳转）
- 异步控制（等待、并发、超时）

当前 banvasgl 的 Renderer/Scene/View 体系和 flow 的 FlowRunner/NodeExecutor 注册表已经在做这件事。它们定义的就是这套统一接口的雏形。

## 各平台适配的职责

每个目标平台只需实现统一接口层的契约，翻译为本平台的 API 调用：

- **Web（当前）**：banvas-runtime 基于 React + Canvas 2D 实现，是第一个也是当前唯一的平台适配
- **macOS/Windows 桌面**：通过 Electron 使用 Web 实现（当前路径），未来可选原生 Metal/D3D 适配
- **iOS**：Swift + Core Animation / Metal
- **Android**：Kotlin + Canvas / Vulkan

每个适配是独立的包/模块，互不依赖，可以按需实现——先做 Web 和 Electron 桌面，移动端按业务需求后续补充。

## banvas-runtime 的定位

banvas-runtime 是当前唯一的平台适配实现（Web/React），也是未来所有平台适配的参考实现。它的风险最高（ADR-018 已说明），因为它的正确性直接决定了构建产物是否符合设计稿预期。

随着统一接口层的定义逐步稳定，banvas-runtime 会自然演化为「Web 平台适配」而非「唯一的运行态」。新平台适配的加入不影响 banvas-runtime 的代码——它们是平级的兄弟包，共享同一套接口契约。

## 替代方案

### 方案 A：自建 DSL + 编译到原生代码

设计一门领域语言，将 UI 和逻辑编译为 Swift/Kotlin/JS 等原生代码。

否决原因：没有人类开发者需要读写这门语言。编译器的开发和维护成本极高，且引入了「编译器 bug」这个新的风险源。对于声明式 UI + 有限逻辑流的场景，编译带来的性能收益远不如解释执行带来的灵活性价值大。

### 方案 B：生成 Flutter/React Native 代码

将 JSON 转换为 Flutter Widget 代码或 RN 组件代码，复用现有跨平台框架。

否决原因：引入了对第三方框架的强依赖，受限于其 API 设计和更新节奏。且 Banyan 的视觉原语是自定义的（基于 banvasgl），与 Flutter/RN 的 Widget 体系不一定对得上，胶水代码的维护成本不可控。

### 方案 C：全平台统一用 WebView

所有目标平台都通过 WebView 跑 Web 版本的 banvas-runtime。

部分采纳作为过渡方案：当前 Electron 桌面版本质上就是这个路径。但 WebView 在移动端的性能和原生体验有明显天花板（内存占用、滚动流畅度、启动速度）。长期来看，高性能场景仍需原生适配。

## 后果

- 跨平台路径明确：JSON 中间表示 + 统一接口 + 平台适配，不走自建语言路线
- banvasgl 和 flow 引擎的核心任务之一是稳定统一接口层的定义——这是所有平台适配的契约基础
- banvas-runtime（Web 适配）作为第一个也是当前唯一的实现，同时承担「参考实现」和「接口验证」的角色
- 新平台的支持是增量式的：定义好接口后，一次加一个平台适配包，互不影响
- Electron 桌面路径在中短期内足够（WebView 跑 Web 实现），原生桌面适配按需推进
- 这个架构天然支持热更新：只要平台适配层不变，更新 JSON 数据即可改变应用行为，无需重新构建安装包
