# ADR-040：XiangDi 知识定义与生命周期

**状态**：已采纳  
**决策日期**：2026-06  
**决策者**：Banyuan 核心团队

---

## 背景

ADR-003 确立了知识的消费模式（Tool 按需检索），ADR-007 确立了知识的检索技术栈（向量+BM25+GraphRAG）。但两者都没有回答一个更根本的问题：**XiangDi 需要的"知识"到底是什么？它和业界 RAG 场景的知识是同一种东西吗？**

业界 RAG 通常处理的是**事实性知识**——公司文档、FAQ、产品手册——LLM 不知道但只需"读取并转述"。XiangDi 的场景更复杂：LLM 需要先理解 BanvasGL 的能力体系（每种组件是什么、能做什么、适合什么场景），然后才能将用户的 UI 意图翻译为合法的结构化 JSON 输出。

缺乏对知识本质的清晰定义，导致以下问题：

1. composition 种子格式不统一（部分使用旧版简化 schema，非当前 AI Projection 格式）
2. 知识的生成策略不明确——哪些应自动生成、哪些需人工 curate、哪些由 LLM 辅助
3. 知识更新与 BanvasGL 版本演进的关联未被形式化
4. Schema 种子只描述了格式约束，缺少语义维度（每种 ViewType 是什么、适合什么场景）
5. "应用级知识"的归属和存储位置不明确

## 决策

### 核心定义

**XiangDi 的知识 = BanvasGL 能力体系的完整认知。**

它包含两个不可分割的维度：

1. **语义维度（What）**：每种 ViewType / graphType / layoutMode 是什么、能做什么、适合什么场景、有什么限制——让 LLM 知道"该用什么"
2. **格式维度（How）**：在理解语义后，如何将其表达为合法的 AI Projection JSON——让 LLM 知道"怎么写"

两个维度缺一不可：没有语义维度，LLM 不知道什么时候该用 `flex` 什么时候该用 `free`；没有格式维度，LLM 知道该用 `flex` 但写不出合法 JSON。

这个定义区别于传统 RAG 的关键在于：传统 RAG 的知识是"LLM 读了之后转述给用户"，XiangDi 的知识是"LLM 读了之后用来**决策**（选什么组件）和**生成**（写出合法 JSON）"。

### 知识的认知链路

当用户说"帮我做一个商品详情页"时，LLM 的认知链路是：

```
用户意图（商品详情页）
    ↓ 语义知识：知道详情页通常包含大图、标题、价格、描述、按钮
    ↓ 语义知识：知道 CombinedView(flex) 适合做纵向排列容器
    ↓ 语义知识：知道 ImageView 适合展示商品图、TextView 适合文字
    ↓ 格式知识：知道 flex 容器的 JSON 怎么写
    ↓ 格式知识：知道子节点的 transform/size/decoration 格式
    ↓
合法的 AI Projection JSON
```

**关键洞察**：LLM 对通用 UI 世界的理解（什么是详情页、什么是导航栏）来自预训练，不需要教。需要教的是 **BanvasGL 特有的能力体系**——哪些是 BanvasGL 能表达的、每种表达方式的能力边界和适用场景。

### 知识归属：系统级 vs 应用级

#### 决策：不存在独立的"应用级知识"层

**appJSON 本身就是应用的全部知识。**

应用的设计风格、布局偏好、颜色体系、间距规范——全都隐含在 appJSON 的页面结构中。不需要额外抽取一份应用级视觉规范单独存储。

LLM 理解当前应用风格的方式是：通过程序化工具从 appJSON 中提取视觉参数摘要（结构匹配 + 统计分析），零 token 消耗地建立 Graph/View 到视觉参数的映射关系，然后把精炼后的摘要给 LLM 进行推理。

这遵循 ADR-035（token 成本意识）原则：**用工程手段代替 token 消耗。**

#### 知识架构全貌

```
knowledge-server（系统级，所有应用共享）：
  Schema 种子 —— BanvasGL 能力认知（语义+格式）
  Composition 种子 —— 系统内置组合模式参考
  Theme 种子 —— 系统默认视觉表现知识（语义映射+层次规则+参数值）
  通过 knowledge_search 工具按需检索

appJSON（应用数据，每个应用私有）：
  页面结构本身隐含了应用的所有视觉决策
  通过程序化工具（analyze_app_style / get_pages）提取和呈现
  零额外存储成本
  跟随应用生命周期，删除即清理
```

#### 为什么不需要独立的应用级知识存储

1. **appJSON 已经包含一切**：颜色、间距、字号、组合方式都在 JSON 结构中
2. **规模小到不需要向量检索**：一个应用的视觉参数摘要可能就几百字节，直接作为工具返回值即可
3. **避免一致性问题**：如果单独存一份应用视觉规范，pages 改了但规范没同步就会出错
4. **缓存友好**：system prompt 不注入应用特定信息，保持前缀稳定，最大化 Prompt Cache 命中率
5. **消费方式统一**：所有知识都走 Tool 按需拉取，概念一致

#### 程序化风格提取工具示例

```typescript
// analyze_app_style 工具（零 token 消耗的工程手段）
// 从 appJSON 中结构匹配 + 统计分析，输出精炼摘要
function analyzeAppStyle(appJSON: SerializedData): StyleSummary {
  const allNodes = flattenAllNodes(appJSON.data.scenes)
  return {
    dominantColors: extractColors(allNodes),      // 按频率排序的主要颜色
    fontSizeScale: extractFontSizes(allNodes),    // 字号梯度
    spacingScale: extractGaps(allNodes),          // 间距梯度
    radiusScale: extractRadiuses(allNodes),       // 圆角梯度
    layoutPreference: countLayoutModes(allNodes), // 布局模式偏好
  }
}
```

LLM 收到的是精炼摘要（如"主色调 #1677FF，正文 14px，间距 8/12/16/24px，布局偏好 flex(column)"），而非原始 JSON dump。

#### 隐私合规边界

```
系统级知识（knowledge-server）：
  ✅ 公共数据，无隐私问题
  ✅ BanvasGL 能力说明 + 系统默认模式 + 默认设计 Token
  ❌ 不存储任何用户/应用特定信息

应用数据（appJSON，banyan 后端）：
  ✅ 用户私有，跟随应用生命周期
  ✅ 受租户隔离机制保护
  ✅ 用户删除应用时级联清理
  ❌ 不进入 knowledge-server 的公共检索池
```

### 与业界 RAG 知识的对比

| 维度 | 传统 RAG 知识 | XiangDi 知识 |
|------|------|------|
| 目的 | 让 LLM 回答它不知道的事实 | 让 LLM 理解能力体系后正确决策和生成 |
| 消费方式 | LLM 读取后用自然语言转述 | LLM 读取后用于选型决策 + JSON 格式产出 |
| 认知链路 | 检索 → 转述 | 检索 → 理解能力 → 选型决策 → 格式化输出 |
| 正确性验证 | 人工判断是否回答准确 | 程序化验证（`fromAIProjection()` 反序列化成功）+ 视觉合理性 |
| 知识来源 | 外部文档、数据库 | 代码类型系统 + 能力说明 + 设计规范 + 人工 curated 模式 |
| 更新频率 | 业务驱动，不可预测 | 版本驱动，随 BanvasGL 发版同步 |
| 切片粒度 | 段落/句子级 | 以"一个 ViewType 的完整描述（语义+格式）"为原子单位 |
| 应用特定知识 | 通常需要独立知识库 | appJSON 本身 + 程序化提取，无需额外存储层 |
| 共同点 | 都是 LLM 上下文窗口装不下的外部信息，按需检索注入 |

### 三层知识分类（系统级）

#### 第一层：能力认知知识（Schema Seeds）

"这是什么 + 怎么写"——每种 BanvasGL 组件的完整认知。

- **本质**：BanvasGL 能力体系的自然语言描述 + AI Projection 格式规范
- **语义维度内容**：
  - 每种 ViewType 的功能定位和适用场景
  - 各 graphType 的视觉语义（RECT 适合什么、ELLIPSE 适合什么）
  - layoutMode 各值的能力边界（flex 能做什么、不能做什么）
  - 属性的业务含义（不只是类型约束，还有"这个属性控制什么视觉效果"）
  - 选型决策指南（什么时候该选、什么时候不该选）
- **格式维度内容**：属性名 + 类型 + 是否必填 + 默认值 + 合法值域 + 最小示例
- **正确性标准**：LLM 参考此知识生成的 JSON 能通过 `fromAIProjection()` 反序列化
- **生成方式**：格式维度从 TypeScript 类型定义自动提取（`generate-knowledge.ts`）；语义维度需要人工编写或从 JSDoc/README 中提取
- **更新策略**：格式维度跟随 BanvasGL 版本自动重建；语义维度在新增/修改 ViewType 能力时同步更新
- **原子单位**：一个 ViewType（如 TEXTVIEW）的完整描述，语义和格式合为一体

示例（CombinedView 的能力认知）：
```markdown
## COMBINEDVIEW — 容器视图

### 是什么
容器视图，类比 HTML 的 <div>，可包含任意子视图。是构建复杂 UI 结构的基础。

### layoutMode 能力
- "free"：自由布局，子视图绝对定位。适合不规则布局、重叠效果、精确像素控制。
- "flex"：弹性布局，子视图沿主轴排列。适合表单、列表项、卡片内部结构。
  支持：direction(row/column), gap, alignItems, justifyContent
  限制：不支持 wrap，不支持嵌套 flex 的百分比尺寸
- "list"：列表布局，纵向滚动的重复项容器。适合数据列表、消息流。
- "grid"：网格布局，按行列排列。适合商品展示、图片画廊。
- "scroll"：滚动容器，内容超出时可滚动。适合长内容页面。

### 选型决策指南
当你需要...                           → 选择...
纵向排列多个元素                       → flex (direction: column)
横向排列等宽按钮组                     → flex (direction: row)
子元素需要重叠或精确定位               → free
展示数量不定的同类项目（商品列表）     → list
展示固定列数的网格（图片墙）           → grid
内容超出容器需要滚动                   → scroll

### JSON 格式
{
  "type": "COMBINEDVIEW",
  "id": "...",
  "transform": { "x": 0, "y": 0 },
  "size": { "width": 375, "height": 600 },
  "layoutMode": "flex",
  "flexLayout": { "direction": "column", "gap": 12 },
  "children": [...]
}
```

#### 第二层：组合模式知识（Composition Seeds）

"怎么组合"——将能力认知应用于具体 UI 场景的最佳实践。

- **本质**：高质量 Few-shot 示例，展示如何运用第一层的能力认知解决具体 UI 问题
- **角色**：系统内置参考。实际对话中视觉智能体会结合用户意图 + 网络搜索 + 系统 Composition 参考，实时生成本轮的组合方案
- **语义维度内容**：解决什么用户场景、为什么选择这种组合方式、有哪些变体
- **格式维度内容**：完整的 AI Projection JSON 片段 + 层次结构说明 + 典型尺寸/间距
- **正确性标准**：示例 JSON 必须通过 `fromAIProjection()` 验证 + 渲染结果视觉合理
- **生成方式**：LLM 基于第一层能力认知生成 → `fromAIProjection()` 程序化验证 → 人工 review 视觉效果
- **更新策略**：新增 UI 模式时人工 curate；版本迭代时用验证脚本批量检查已有模式是否仍然合法
- **原子单位**：一个 UI 组合模式（如 login-form）的完整描述

#### 第三层：视觉表现知识（Theme Seeds）

"长什么样"——结构确定后，如何赋予视觉表现。

- **本质**：视觉决策的知识化表达——不仅是参数值，更是"什么场景用什么视觉处理"的决策逻辑
- **角色**：系统默认参考。用户未指定风格时使用默认主题；用户指定风格偏好时，LLM 结合系统 Theme 参考 + 用户意图 + 网络搜索生成新的视觉方案
- **内容**（三个层次）：
  - **语义映射**：场景到视觉处理的对应关系（主操作 → 品牌色填充、错误状态 → 红色边框+浅红背景、禁用 → 灰色+降低透明度、悬浮卡片 → 阴影+白底）
  - **层次规则**：视觉层级如何通过字号/字重/颜色对比度表达（标题 vs 正文 vs 辅助文字的区分逻辑、信息主次的视觉权重分配）
  - **参数值**：上述规则的具体实例化（品牌色=#1677FF、正文=14px、基础间距=8px）
- **与 Composition 的递进关系**：Composition 告诉 LLM "登录表单由容器+输入框+按钮组成"，Theme 告诉 LLM "这个按钮应该用品牌色填充、圆角 8px、字号 16px 白色，因为它是主操作"
- **正确性标准**：值域合法（颜色为有效 hex、数值在合理范围内）+ 语义映射与实际视觉效果一致（需人工判断）
- **生成方式**：设计师/产品维护，纯手工——视觉语义映射背后是设计决策，不可自动生成
- **更新策略**：设计规范变更时更新

### 三层之间的关系

三层是"能力 → 结构 → 表现"的层层递进：

```
第一层（能力认知）：我有哪些积木，每块积木能做什么
        ↓ 选型决策
第二层（组合模式）：积木怎么搭成用户要的结构
        ↓ 视觉决策
第三层（视觉表现）：搭出来的结构用什么视觉表现
```

每一层解决上一层留下的问题：Schema 解决"用什么"，Composition 解决"怎么搭"，Theme 解决"长什么样"。缺任何一层，LLM 的输出都不完整。

第二层和第三层都是"系统提供参考，实际生产时结合应用上下文动态决策"的模式。

### 知识质量的程序化验证原则

与传统 RAG 知识依赖人工评估不同，XiangDi 知识的正确性可以部分程序化定义：

1. **Schema 种子格式维度**：从类型系统生成，类型系统本身就是 ground truth
2. **Schema 种子语义维度**：需要人工 review 确认描述与实际能力一致（不可自动化）
3. **Composition 种子**：`fromAIProjection(seed.example)` 必须成功反序列化（格式正确），且渲染结果视觉合理（语义正确，需人工判断）
4. **Theme 种子**：值域校验（Zod schema 或简单类型检查）

CI 流水线可以拦截格式维度的错误；语义维度的正确性依赖 code review。

### 为什么不需要完整 RAG 流程

XiangDi 的知识有三个本质区别于传统 RAG 场景的特征，使得我们只需参考 RAG 的检索-注入范式，而无需经历完整的"获取→清洗→切片→向量化"管线：

1. **知识源头是自己的代码和能力设计**：Schema 种子的格式维度从类型定义自动生成，语义维度从开发者的设计意图提取——源头精确、无噪声
2. **知识规模有限且可枚举**：8 种 AI Projection 节点类型 + 有限组合模式 + 有限设计 Token，每个版本知识总量确定
3. **格式正确性可程序化定义**：`fromAIProjection()` 是格式验证的绝对标准

## 冷启动与热更新策略

冷启动必须自底向上——第一层是第二层的前提，第二层是 LLM 正确生成的基础。每层内先说冷启动（系统上线前准备），再说热更新（系统运行后持续演进）。

### 第一层：Schema Seeds（能力认知）

**现状**：`generate-knowledge.ts` 脚本内硬编码 Markdown 内容。已包含语义描述（描述、使用场景）和格式说明（属性结构、JSON 示例），但语义描述的"决策导向"深度不足。

**冷启动**：

当前 10 个 Schema 种子已生成，内容已包含语义+格式双维度。需要语义增强——为每种 ViewType 补充"选型决策指南"，告诉 LLM 什么时候该选、什么时候不该选：

```markdown
### 选型决策指南
当你需要...                    → 选择...
纵向排列多个元素               → flex (direction: column)
横向排列等宽按钮组             → flex (direction: row)
子元素需要重叠或精确定位        → free
```

这类决策导向的语义描述只能人工编写——因为只有设计者知道每种能力的边界和适用场景。

语义内容维护在 `generate-knowledge.ts` 脚本中。种子数量有限（10 个），修改语义描述只需编辑一个文件，改完 build 即生效。

**热更新**：

触发条件 A（格式变更）：BanvasGL 新增/修改 ViewType 属性 → 修改 `generate-knowledge.ts` → build 自动重建种子。

触发条件 B（语义不足，错误驱动）：LLM 做出错误选型决策 → 分析原因 → 如果是知识描述不够清晰 → 增强对应 ViewType 的语义描述 → rebuild。

```
LLM 产出错误 → fromAIProjection() 失败 or 视觉不合理
    ↓ 分析错误原因
    ├── LLM 能力问题（换 prompt/换模型）→ 不修改知识
    ├── 格式描述不清（属性含义模糊）→ 增强格式维度
    └── 选型错误（不该用 free 用了 free）→ 增强语义维度（选型决策指南）
```

### 第二层：Composition Seeds（组合模式）

**现状**：8 个种子文件使用旧格式（与当前 AI Projection 不兼容），需要全部重建。

**冷启动**：

冷启动阶段没有用户使用数据，团队基于经验预设一批覆盖高频 UI 场景的种子。

Step 1 — 确定初始覆盖范围（5~8 个典型 UI 模式）：

- 登录/注册表单（form 类）
- 列表页（list + 列表项）
- 详情页/信息卡片（纵向信息展示）
- 仪表盘/统计卡片（grid 布局 + 数据展示）
- 导航结构（顶部导航 / 侧边导航）
- 模态对话框（浮层 + 按钮组）
- 数据表格（表头 + 行列结构）

Step 2 — 从编辑器导出：在 Banyan 编辑器中手动制作目标 UI → `toAIProjection()` 导出为 AI Projection JSON。来源于编辑器的 JSON 天然合法（能渲染就是合法的）。

Step 3 — LLM 标注语义 + 人工 review：让 LLM 分析已有的合法结构，标注场景说明、选型决策理由、常见变体。精简到"最小充分结构"——去除具体 uuid，替换为语义化 id。人工确认标注质量后入库。

**热更新**：

系统运行后积累了物料使用量数据，热更新切换为数据驱动模式——高使用量 = "这个组合模式有普遍价值"的客观信号，比人为 curate 更准确、更不容易有盲区。

```
物料使用量统计 → 筛选 top N（排除已有种子）→ toAIProjection() 导出
    → LLM 标注语义（场景/决策理由/变体）→ 人工 review 标注质量 → 入库
```

触发条件 A（新高频物料涌现）：定期（或阈值触发）扫描物料使用量，新进入 top N 的物料走导出 + LLM 标注 + review 流程补充为种子。

触发条件 B（版本升级）：BanvasGL 升级导致 AI Projection 格式变化 → CI 验证脚本批量检查所有 composition 种子 → 失败的种子需人工修复。

触发条件 C（缺口驱动）：用户反馈 XiangDi 不会做某类页面 → 如果物料库中有相关物料直接走导出流程，否则先在编辑器中制作再走流程。

### 第三层：Theme Seeds（视觉表现）

**冷启动**：

设计师将设计规范中的决策逻辑显式化——不仅写"品牌色是 #1677FF"，更要写"什么时候该用品牌色"。content 字段应包含完整的视觉决策知识（让向量检索能命中语义查询）：

```markdown
# 默认主题 — 视觉表现知识

## 语义映射（什么场景用什么视觉处理）

### 操作层级
- 主操作（确认/提交/下一步）→ 品牌色填充 #1677FF + 白色文字 + 圆角 8px
- 次操作（取消/返回）→ 描边样式 border: 1px solid #D9D9D9 + 深色文字
- 危险操作（删除/清空）→ 红色填充 #FF4D4F + 白色文字
- 禁用状态 → 灰色背景 #F5F5F5 + 浅灰文字 #00000040

### 反馈状态
- 成功 → 绿色 #52C41A（图标/边框/背景任选）
- 错误 → 红色 #FF4D4F + 浅红背景 #FFF2F0
- 警告 → 橙色 #FAAD14 + 浅橙背景 #FFFBE6
- 信息提示 → 品牌色 #1677FF + 浅蓝背景 #E6F4FF

### 视觉层级
- 一级标题 → 24px Bold #000000E0
- 二级标题 → 18px Semibold #000000E0
- 正文 → 14px Regular #000000E0
- 辅助文字 → 12px Regular #00000073
- 占位文字 → 14px Regular #00000040

## 层次规则（如何表达信息主次）

- 通过字号梯度区分层级：xs 12px → sm 14px → base 16px → lg 18px → xl 24px
- 通过颜色对比度区分重要性：主要 E0 → 次要 73 → 辅助 40（透明度后缀）
- 通过间距梯度创造分组：相关元素紧凑(8px)，不同分组拉开(16~24px)

## 参数值（上述规则的实例化）

- 颜色体系：primary #1677FF | success #52C41A | error #FF4D4F | warning #FAAD14
- 间距梯度：xxs 2px | xs 4px | sm 8px | md 12px | lg 16px | xl 24px | xxl 32px
- 圆角梯度：xs 2px | sm 4px | md 6px | lg 8px | full 9999px（胶囊形）
```

**热更新**：

触发条件：设计规范变更（频率极低）。流程：设计师更新视觉决策 → 开发者同步 content → CI 值域校验 → 合入后 upsert 到知识服务。

### 执行路线图

```
Phase 0（立即可做）：
  □ Theme 种子：视觉决策知识写入 content 字段，优化向量检索命中
  □ Schema 种子：审视现有语义描述质量，确认基线

Phase 1（冷启动核心）：
  □ Schema 种子语义增强：为每种 ViewType 补充选型决策指南
  □ Composition 种子重建：团队预设 5~8 个典型 UI 模式 → 编辑器制作 → 导出 + LLM 标注 → 替换旧格式种子
  □ CI 验证脚本：fromAIProjection() 批量验证所有 composition 种子

Phase 2（验证+迭代）：
  □ 端到端测试：用 XiangDi 生成典型页面，观察选型和输出质量
  □ 错误驱动迭代：根据错误输出反馈增强 Schema 语义描述
  □ 补充 Composition 种子覆盖度（根据实际使用中暴露的缺口）

Phase 3（热更新常态化）：
  □ 错误驱动的知识增强成为日常流程
  □ 版本升级自动验证（CI 拦截失效种子）
  □ LLM 辅助扩展 Composition（有足够黄金样本后）
```

## 后果

### 正面影响

- 明确了知识的双维度定义（语义+格式），避免只教 LLM "怎么写"而不教"是什么"
- 消除了"应用级知识存哪里"的困惑——appJSON 本身 + 程序化提取，无需额外存储层
- system prompt 保持稳定，最大化 Prompt Cache 命中率
- 所有知识消费走 Tool 模式，概念统一
- Schema 种子有了更高的质量要求——不仅包含类型约束，还必须包含能力说明和选型指南
- Composition 种子有了明确的格式标准和生成路径
- 建立了错误驱动的知识增强闭环
- 隐私边界清晰——系统知识公共，应用数据私有

### 负面影响 / 权衡

- Schema 种子的语义维度需要人工编写，有维护成本（但规模有限，只有 10 个种子）
- Composition 种子冷启动需要团队手动在编辑器中制作初始样本，有一定人力投入
- 程序化风格提取工具（analyze_app_style）需要开发和维护
- 依赖 LLM 正确调用工具来获取知识——如果 LLM 跳过检索直接生成，质量可能下降

### 行动项

- 统一现有 composition 种子为 AI Projection 格式（当前部分使用旧格式）
- 增强 Schema 种子的语义维度——为每种 ViewType/layoutMode 补充能力说明、适用场景和选型决策指南
- 在 `generate-knowledge.ts` 中补充 CI 验证步骤（格式维度）
- 开发 `analyze_app_style` 程序化工具，从 appJSON 中提取风格摘要
- 团队预设 5~8 个 Composition 初始种子（编辑器制作 → 导出 → LLM 标注）；热更新阶段建立物料使用量驱动的选种机制
- Theme 种子的 content 字段优化——将 token 数据写入 content 供向量检索命中
