# ADR-005: MVP 阶段选择 Canvas 2D 而非 WebGPU

**状态**: 已采纳（计划在后续版本迁移到 WebGPU）
**日期**: 2026-05-14
**决策者**: 陈班

## 背景

BanvasGL 需要一个渲染后端。候选方案有 Canvas 2D、WebGL 2、WebGPU。项目目标是快速验证产品方向，同时保留后续性能升级的可能性。

## 决策

MVP 阶段使用 Canvas 2D 双缓冲渲染，将渲染层抽象为 `Renderer` 接口，后续可替换为 WebGPU 实现。

## 考虑过的方案

**方案 A（采纳）：Canvas 2D** — API 简单成熟，文本渲染开箱即用，无需 shader 开发，双缓冲即可满足当前性能需求。缺点：大量节点时性能有上限，无法利用 GPU 加速。

**方案 B：WebGL 2** — 性能好，但文本渲染需要自己实现（MSDF/SDF font atlas），开发成本高。且 WebGL 在 Safari 上的实现有各种坑。

**方案 C：WebGPU** — 性能最优，现代 API 设计。但浏览器支持尚未普及（尤其 Firefox/Safari），且开发 2D 渲染管线的工程量大。列入路线图中的后续目标。

## 后果

- MVP 可以快速完成渲染层开发，专注于产品逻辑验证
- 文本渲染、图片绘制等能力开箱可用
- 当前的 Renderer 接口层为后续切换到 WebGPU 预留了可能（但实际迁移仍需大量工作）
- 节点数 > 1000 时可能出现渲染性能瓶颈（当前业务场景下不太会触及）
- 重计算已分流到 Web Worker，不阻塞渲染主线程
