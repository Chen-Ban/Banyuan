关系说明：

- P1→P2：工程化优先原则使得执行型 SubAgent 采用 tool_use 循环而非带冗余 Thought 的 ReAct

- P1→P4：工程化优先确立"知识用于程序化验证"的基调——知识不是给人读的转述，是给代码验证的决策依据（P4 同时 refines engine:A0——知识范畴对应运行时机制层）

- P1→P8：工程化优先要求明确的分层职责——每层只重试自己能解决的问题，避免重试爆炸

## 执行哲学

### P1. 工程化优先：确定性逻辑交给代码，LLM 只做语义推理

**✅ 已实施**

能用代码确定性解决的问题不交给 LLM。LLM 的价值在语义理解和创造性决策，不在执行确定性逻辑。

**决策链：** 早期方案倾向于用 SubAgent 做各种判断（格式验证、引用完整性检查、状态路由），但 LLM 在确定性任务上既慢又贵且不稳定。审视每个 LLM 调用点后发现：格式验证可以用 Zod schema 程序化完成、引用完整性可以用图遍历程序化校验、无历史状态时的路由可以直接 if 判断。将这些确定性逻辑从 LLM 抽离后，Agent 系统更稳定、更快、更便宜。

**约束：**

- 格式校验用 Zod schema，不用 LLM 判断输出是否合法
- 引用完整性（callFlow→函数ID、sideEffect→集合名）用程序化图遍历，审计节点零 token 校验
- 状态机转移用显式 PHASE_TRANSITIONS map，不用 LLM 判断下一个 phase
- intent 节点在无历史状态时程序化直出，不调 LLM
- SubAgent 内部的 output_validation 失败通过追加修正 prompt 重试，不上浮到 Orchestrator 仲裁
- 工具是确定性操作（执行 Agent 已决策好的动作），不在工具内部再做 LLM 决策
- commit 节点纯程序化（零 token）——只做数据组装和事件推送

**反例：**

- 工具里藏 LLM（generate_cloud_function 内部发起独立 LLM 调用）——违背工具的本质定位，子 LLM 丢失主 Agent 的完整上下文。改为后端 Worker 自身在 think 阶段生成 FlowSchema，工具只做纯写入

### P2. Agentic Loop：tool_use 循环而非经典 ReAct

**✅ 已实施** · 由 P1 驱动

执行型 SubAgent 采用 think↔tools 多轮循环模式，而非 Thought-Action-Observation 的经典 ReAct 三段式。

**决策链：** 经典 ReAct 要求 Agent 在每一步显式输出 Thought（自然语言推理过程），这在 token 消耗和延迟上代价高昂。现代 LLM（DeepSeek/Claude）的 function calling 能力已内化推理过程——LLM 直接输出 tool_call 决策，无需中间 Thought 文本。Agentic Loop 的模式是：LLM 输出 tool_call → 执行工具 → 将结果追加到 messages → 再次调用 LLM，循环直到 LLM 决定不再调用工具（输出最终文本）。

**约束：**

- 执行型 SubAgent 有 maxIterations 循环上限（默认 15）
- 每轮循环：LLM 可输出 0~N 个 tool_call（并行调用），执行后结果统一追加
- 终止条件：LLM 不输出 tool_call（表示任务完成）或达到 maxIterations
- 规划型 SubAgent 是 Agentic Loop 的退化形式（通常 0~2 轮工具调用做信息采集，核心是一次结构化输出）

---

### P3. 全流程自动执行，验收后移

**✅ 已实施** · 由 P1 驱动

零代码用户不是开发者，给用户看中间产物（契约/方案）没有意义。全流程自动执行，验收点在最终结果交付之后。

**决策链：** 早期设计有 humanGate（规划阶段产出方案后暂停等待用户确认再执行）。但目标用户是零代码用户——看不懂数据表 Schema、不理解 FlowSchema 节点图、无法评估 UI 设计规格的技术文档。中间确认只增加等待时间，不增加质量。用户真正能评估的是最终效果（画布上的可视化结果）。因此去掉所有中间确认，自动执行到底，用户在 awaiting_confirm 状态下查看预览效果后一次性验收。

**约束：**

- awaiting_confirm 是唯一用户决策点
- 用户不满意发新消息修正（新 Dialogue），不通过 resume 机制
- 内部回退对用户不可见（building phase 封装了全部内部复杂性）
- awaiting_confirm 无 TTL 清理，用户可跨越任意时间间隔回来确认或拒绝

## 知识哲学

### P4. 知识本质：不是转述，是决策和生成

**✅ 已实施** · 由 P1 驱动 · refines engine:A0

XiangDi 的知识与传统 RAG 本质不同。传统 RAG 的知识是"LLM 读了之后转述给用户"——知识质量取决于回答是否准确。XiangDi 的知识是"LLM 读了之后用来决策和生成合法 JSON"——知识质量取决于生成的 JSON 能否通过程序化验证器。

> **A0 机制/策略定位：** banvasgl 作为图形运行时只提供机制，知识的范畴严格对应运行时暴露的机制层能力。三个领域的知识内容——ViewType 渲染/布局机制、FlowNode 执行机制、序列化机制——都是运行时的 mechanism 定义；策略层（编辑状态机归 banyan、运行态高级交互识别归 banvas-runtime）不入库，因为策略由上层注入而非 banvasgl 提供。

**知识定义：** XiangDi 的知识 = 生成完整应用所需的全部认知，即 BanvasGL 图形运行时机制层能力的完整映射。具体覆盖三个领域：

- **UI 领域**：ViewType 的能力、参数、组合方式、布局机制、视觉风格
- **Flow 领域**：FlowNode 的类型、连接规则、触发条件、事件绑定机制
- **Data 领域**：CollectionField 的类型、索引策略、集合命名规范、云函数签名

每个领域的知识包含两个维度：

- **语义维度**（是什么、适合什么场景）——告诉 LLM "什么场景该选什么能力"：功能定位、适用场景、选型决策指南、组合模式、最佳实践。语义维度需要领域知识和判断力，不可自动化生成。
- **格式维度**（怎么写出合法 JSON）——告诉 LLM "选定能力后怎么写出合法结构"：参数结构、类型约束、必填/可选字段、取值枚举、JSON 示例。格式维度本质是类型定义的另一种表达，可从 TypeScript 定义自动提取。

**三层递进：**

- **Primitive 种子**（原子能力认知）：单个 ViewType / FlowNode / FieldType 的完整认知（语义 + 格式），是知识体系的最小不可分割单元。如"TextView 是什么、参数结构如何、什么场景用它"
- **Composition 种子**（组合模式）：多个 Primitive 如何协作完成典型场景。包括同领域组合（如 CombinedView + TextView + ImageView 构成卡片）、跨领域的 bindflow 绑定模式（如 onClick 事件绑定 navigate FlowNode）、fullstack 全栈拆解（如"表单提交"拆解为 UI 表单 + Flow 节点图 + Data 集合 + 云函数）
- **Convention 种子**（惯例约定）：视觉约定（间距/圆角/配色惯例）、流程约定（错误处理统一模式）、数据约定（集合命名规范）

**正确性标准（程序化验证）：**

- UI 正确性：`fromAIProjection()` 能否成功将 AI 生成的 JSON 还原为 Scene 实例（无异常 + 属性完整）
- Flow 正确性：FlowSchema Zod 验证通过（节点类型合法、边连接合法、无悬空引用）
- Data 正确性：CollectionSchema Zod 验证通过 + 云函数签名 Zod 验证通过

**知识归属：**

- **系统级知识**存 knowledge-server（公共，所有应用共享）——BanvasGL 引擎能力认知，与具体应用无关
- **应用级知识** = 应用数据本身（appJSON + CollectionSchema + CloudFunctions）——通过程序化工具（如 list_pages、get_page_projection、list_collections）按需提取摘要，无需额外存储层

所有知识消费走 Tool 模式（SubAgent 通过 knowledge_search 工具按需检索），system prompt 不注入应用特定信息，保持 Prompt Cache 命中率。

**决策链：** XiangDi 产出的是结构化 JSON 而非自然语言 → 产出必须通过格式验证器 → 知识的价值在于帮助 LLM 生成合法且合理的结构 → 知识质量的衡量标准是"用了这条知识后生成结果的验证通过率" → 这决定了知识的组织方式（按验证器能检查的维度来切分）和质量保障方式（CI 验证格式维度，Code Review 验证语义维度）。

**约束：**

- 每个种子必须注明其正确性验证方式（哪个验证器、什么输入、期望什么结果）
- 语义维度的正确性依赖 Code Review（人工判断"选型指南是否合理"）
- 格式维度的正确性可 CI 自动验证（自动生成 → diff 对比 → 有差异则 fail）
- 三层种子的关系：Composition 引用 Primitive（不重复定义原子能力），Convention 是 Composition 的约束上界（惯例约束组合的自由度）

**反例：**

- 把整篇 API 文档原文塞入知识库——传统 RAG 思路，对结构化生成无效（太散、太冗余、无选型指导）
- 只有格式没有语义——LLM 知道参数怎么填，但不知道什么场景该选哪个组件
- 知识不按版本隔离——旧版本的 ViewType 参数结构可能导致生成结果在新版本验证失败

### P5. 格式维度自动生成，语义维度 LLM 生成 + 人工审批

**✅ 已实施** · 由 P4 驱动 · 投影边界见 schema 域 M4

Primitive Seeds 的格式维度（参数结构、类型约束、必填/可选字段、JSON 示例）由基础库序列化模块产出**全量快照**自动生成；其作为知识被 Agent 消费时须经投影收敛（详见 schema 域 M4）。语义维度（功能定位、适用场景、选型决策指南）由 LLM 在 CI/CD 流程中生成草稿，**经人工审批确认后**才合并入种子。

**决策链：** 格式信息本质上就是类型系统的另一种表达 → 序列化模块的全量快照自动产出，保证与代码同步 → 语义信息需要领域判断（"什么场景该用什么"）→ LLM 可生成初稿但质量不可盲信 → 程序化验证只能验格式不能验"选型是否合理" → 因此语义维度走"LLM 生成 + 人工审批"半自动模式，把质量门留给人工 Code Review。

**约束：**

- 格式维度 = 序列化全量快照，生成器跟随基础库 CI/CD 自动运行；快照入库不投影，消费侧投影（M4）
- 语义维度由 LLM 在 CI/CD 中生成草稿，必须经人工 review/审批后才发送到 knowledge 服务（不直接入库）
- 格式维度可 CI 自动验证（重新生成 → diff 对比 → 有差异则 fail）；语义维度的质量门是人工审批

### P6. 知识按版本隔离，不做跨版本兼容

**✅ 已实施** · 由 P4 驱动

knowledge-server 按 BanvasGL 版本维护独立向量表。引擎升级时创建新表并写入全量种子，旧表保留供回滚。不在同一张表内做版本兼容。

**决策链：** 不同版本的引擎，ViewType/FlowNode/FieldType 可能有增删 → 旧版本知识对新版本可能产生误导 → 按版本隔离确保 AI 检索到的永远是当前版本的正确知识。

**约束：**

- 表名格式：knowledge_v{banvasglVersion}
- XiangDi 服务请求 knowledge-server 时必须带版本号参数
- 保留最近 3 个版本的旧表，超出自动清理
- 回滚 XiangDi 服务版本时自动切回对应版本的知识表

### P6a. 知识冷启动与热更新策略

**✅ 已实施** · 补充 P4~P6

冷启动必须自底向上——第一层（Primitive）是第二层（Composition）的前提，第二层是 LLM 正确生成的基础。每层内先冷启动（系统上线前准备），再热更新（系统运行后持续演进）。

**三层各自的冷启动路径：**

- **Primitive Seeds（能力认知）**：格式维度由基础库序列化模块产出全量快照自动生成；语义维度由 LLM 生成"选型决策指南"草稿（什么时候该选、什么时候不该选），经人工审批后合并入种子。冷启动核心是语义增强——保证每种 ViewType/FlowNode 的选型语义经人工把关后入库。种子 JSON 落知识服务本地（`apps/knowledge-server/seeds/`）作事实源，导入向量库即生效。
- **Composition Seeds（组合模式）**：冷启动阶段没有用户使用数据，团队基于经验预设 5~8 个覆盖高频 UI 场景的种子（登录表单/列表页/详情卡片/仪表盘/导航结构/模态对话框/数据表格）。制作流程：编辑器手动制作目标 UI → `toAIProjection()` 导出合法 JSON → LLM 标注语义（场景说明/选型决策理由/常见变体）→ 精简为最小充分结构 → 人工确认后入库。来源于编辑器的 JSON 天然合法（能渲染就是合法的）。
- **Convention Seeds（惯例约定）**：设计师将设计规范中的决策逻辑显式化——不仅写"品牌色是 #1677FF"，更要写"什么时候该用品牌色"（语义映射）。content 包含完整的视觉决策知识以提高向量检索命中率。

**三层各自的热更新触发条件：**

- **Primitive 热更新**：触发条件 A（格式变更）——BanvasGL 新增/修改 ViewType 属性 → 基础库 CI/CD 重新产出序列化快照 → 重建格式维度。触发条件 B（错误驱动）——LLM 做出错误选型决策 → 分析原因 → 如果是知识描述不够清晰 → LLM 重生成语义草稿 → 人工审批后更新。关键判断：LLM 能力问题（换 prompt/模型）≠ 知识问题，不应盲目修改知识。
- **Composition 热更新**：切换为数据驱动模式——高使用量 = "这个组合模式有普遍价值"的客观信号。触发条件 A（新高频物料涌现）——定期扫描物料使用量，top N 新物料走导出+标注+review 流程。触发条件 B（版本升级）——AI Projection 格式变化 → CI 验证批量检查所有种子 → 失败的需人工修复。触发条件 C（缺口驱动）——用户反馈 XiangDi 不会做某类页面 → 补充对应种子。
- **Convention 热更新**：触发条件为设计规范变更（频率极低）——设计师更新视觉决策 → 开发者同步 content → CI 值域校验 → upsert 到知识服务。

**约束：**

- 冷启动顺序严格：先 Primitive 完整覆盖 → 再 Composition 引用 Primitive → Convention 约束 Composition 的自由度
- 格式维度的冷启动和热更新完全自动化（脚本生成 + CI 验证）
- 语义维度的冷启动和热更新依赖人工（领域知识 + Code Review）
- 热更新永远基于信号驱动（错误反馈/使用量数据/版本升级/缺口反馈），不做无目的的"定期更新"

## 稳定性哲学

### P7. 知识服务降级不致命

**✅ 已实施** · 由 P1 驱动

knowledge-server 不可用时，知识检索工具降级返回空结果，不阻塞主流程。知识是"锦上添花"而非"必须有"。

**实现：** `apps/xiangdi-server/src/knowledge/RemoteKnowledgeStore.ts`（try-catch 降级返回空数组）

**决策链：** knowledge-server 是独立微服务，可能因部署、网络、资源等原因暂时不可用。如果知识不可用就让整个 AI 流程失败，可用性太脆弱。核心认知：LLM 基于预训练知识仍然可以生成应用（只是不了解系统特有约束），知识检索提升生成质量但不是前置必要条件。因此知识服务降级时 Agent 仍可正常执行，只是生成质量可能略降。

**约束：**

- RemoteKnowledgeStore 网络异常时 catch 后返回空结果，不抛出异常
- 降级时 Agent 继续执行，不中断管线
- 降级行为对 LLM 透明——knowledge_search 工具返回"未找到相关知识"，LLM 基于自身能力继续生成
- 生产环境应监控降级频率，频繁降级说明知识服务需要关注

**反例：**

- 知识服务不可用时 Agent 返回错误——可用性过度依赖子服务
- 降级时使用缓存的旧知识——过期知识可能产生误导，不如没有

### P8. SDK 重试与业务重试分层

**✅ 已实施** · 由 P1 驱动

重试职责严格分层：底层 SDK 处理网络重试（HTTP 级），ToolRegistry 处理瞬时业务错误（工具执行级），LLM 自修正处理逻辑错误（推理级）。三层各司其职，避免重试爆炸。

**决策链：** 系统中有三类可恢复错误：(1) HTTP 网络层（连接中断、DNS 超时）→ SDK 内部重试最合适（有完整 HTTP 上下文）；(2) 工具执行瞬时故障（下游服务 429/5xx）→ ToolRegistry 指数退避最合适（知道是哪个工具、可以等待后重试）；(3) LLM 输出不合法（格式错误、引用不存在）→ 追加错误信息让 LLM 在下一轮 think 中修正最合适（只有 LLM 能改变决策）。如果不分层，某个层级会做"不是自己职责"的重试，导致 3×3×3 = 27 次无效重试。

**约束：**

- SDK 层（OpenAI SDK maxRetries=2）：处理网络超时、连接中断、TLS 错误
- ToolRegistry 层（指数退避 maxRetries=3）：处理 429 限流、5xx 服务端暂时故障、socket hang up
- LLM 自修正层（SubAgent 内部追加 error message 重试 <= 3 次）：处理 output_validation 失败、工具逻辑错误
- 三层之间不重叠——ToolRegistry 只在 SDK 最终抛出错误后介入；LLM 只看到 ToolRegistry 最终的 is_error=true
- 所有层级都有硬上限（maxRetries），所有重试最终都会 fail-fast

**反例：**

- 单层统一重试（所有错误在一个地方重试 N 次）——无法区分"等一下就好"和"需要修改参数"
- 无上限重试——某些"瞬时"故障可能持续数分钟，无上限会阻塞用户请求
- LLMRouter 重试 + SDK 重试 + ToolRegistry 重试三者叠加——重试爆炸，单次错误可能触发 2×3×3=18 次请求
