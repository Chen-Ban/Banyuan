关系说明：

- P1→P2：工程化优先原则使得执行型 SubAgent 采用 tool_use 循环而非带冗余 Thought 的 ReAct

- P1→P4：工程化优先确立"知识用于程序化验证"的基调——知识不是给人读的转述，是给代码验证的决策依据

## 执行哲学

### P1. 工程化优先：确定性逻辑交给代码，LLM 只做语义推理

**✅ 已实施**

能用代码确定性解决的问题不交给 LLM。LLM 的价值在语义理解和创造性决策，不在执行确定性逻辑。

**决策链：** 早期方案倾向于用 SubAgent 做各种判断（格式验证、引用完整性检查、状态路由），但 LLM 在确定性任务上既慢又贵且不稳定。审视每个 LLM 调用点后发现：格式验证可以用 Zod schema 程序化完成、引用完整性可以用图遍历程序化校验、无历史状态时的路由可以直接 if 判断。将这些确定性逻辑从 LLM 抽离后，Agent 系统更稳定、更快、更便宜。

**约束：**

- 格式校验用 Zod schema，不用 LLM 判断输出是否合法
- 引用完整性（callFlow->函数ID、sideEffect->集合名）用程序化图遍历，审计节点零 token 校验
- 状态机转移用显式 transition map，不用 LLM 判断下一个 phase
- intent 节点在无历史状态时程序化直出，不调 LLM
- SubAgent 内部的 output_validation 失败通过追加修正 prompt 重试，不上浮到 Orchestrator 仲裁
- 工具是确定性操作（执行 Agent 已决策好的动作），不在工具内部再做 LLM 决策

**反例：**

- 工具里藏 LLM（generate_cloud_function 内部发起独立 LLM 调用）——违背工具的本质定位，子 LLM 丢失主 Agent 的完整上下文。改为后端 Worker 自身在 think 阶段生成 FlowSchema，工具只做纯写入

### P2. Agentic Loop：tool_use 循环而非经典 ReAct

**✅ 已实施** · 由 P1 驱动

执行型 SubAgent 采用 think<->tools 多轮循环模式，而非 Thought-Action-Observation 的经典 ReAct 三段式。

**决策链：** 经典 ReAct 要求 Agent 在每一步显式输出 Thought（自然语言推理过程），这在 token 消耗和延迟上代价高昂。现代 LLM（DeepSeek/Claude）的 function calling 能力已内化推理过程——LLM 直接输出 tool_call 决策，无需中间 Thought 文本。Agentic Loop 的模式是：LLM 输出 tool_call -> 执行工具 -> 将结果追加到 messages -> 再次调用 LLM，循环直到 LLM 决定不再调用工具（输出最终文本）。

**约束：**

- 执行型 SubAgent 有 maxIterations 循环上限
- 每轮循环：LLM 可输出 0~N 个 tool_call（并行调用），执行后结果统一追加
- 终止条件：LLM 不输出 tool_call（表示任务完成）或达到 maxIterations
- 规划型 SubAgent 是 Agentic Loop 的退化形式（通常 0~2 轮工具调用做信息采集，核心是一次结构化输出）

### P3. 全流程自动执行，验收后移

**✅ 已实施** · 由 P1 驱动

零代码用户不是开发者，给用户看中间产物（契约/方案）没有意义。全流程自动执行，验收点在最终结果交付之后。

**决策链：** 早期设计有 humanGate（规划阶段产出方案后暂停等待用户确认再执行）。但目标用户是零代码用户——看不懂数据表 Schema、不理解 FlowSchema 节点图、无法评估 UI 设计规格的技术文档。中间确认只增加等待时间，不增加质量。用户真正能评估的是最终效果（画布上的可视化结果）。因此去掉所有中间确认，自动执行到底，用户在 awaiting_confirm 状态下查看预览效果后一次性验收。

**约束：**

- awaiting_confirm 是唯一用户决策点
- 用户不满意发新消息修正（新 Dialogue），不通过 resume 机制
- 内部回退对用户不可见（building phase 封装了全部内部复杂性）
- awaiting_confirm 无 TTL 清理，用户可跨越任意时间间隔回来确认或拒绝

## 知识哲学

### P4. 知识本质：不是转述，是决策和生成

**✅ 已实施** · 由 P1 驱动

XiangDi 的知识与传统 RAG 本质不同：传统 RAG 是"LLM 读了之后转述"，XiangDi 的知识是"LLM 读了之后用来决策和生成合法 JSON"。知识的正确性标准是程序化验证（fromAIProjection / FlowSchema 验证 / Zod 校验），不是人工判断是否回答准确。

**决策链：** XiangDi 产出的是结构化 JSON 而非自然语言 -> 产出必须通过格式验证器 -> 知识的价值在于帮助 LLM 生成合法且合理的结构 -> 知识质量的衡量标准是"用了这条知识后生成结果的验证通过率"。

**约束：**

- 每个种子必须注明其正确性验证方式（哪个验证器、什么输入、期望什么结果）
- 语义维度（描述和选型指南）的正确性依赖 Code Review
- 格式维度（JSON schema 示例）的正确性可 CI 自动验证

---

### P5. 格式维度自动生成，语义维度人工编写

**✅ 已实施** · 由 P4 驱动

Primitive Seeds 的格式维度（参数结构、类型约束、必填/可选字段、JSON 示例）从 TypeScript 类型定义自动提取。语义维度（功能定位、适用场景、选型决策指南）由人工编写。

**决策链：** 格式信息本质上就是类型定义的另一种表达 -> 自动生成保证与代码同步 -> 语义信息需要领域知识和判断力（"什么场景该用什么"）-> 不可自动化 -> 分层处理各司其职。

**约束：**

- 格式生成器脚本放在对应包的 scripts/ 目录
- 格式生成器跟随包的 postbuild 钩子自动运行
- 语义维度变更不走自动化流程，直接编辑种子文件并提交

---

### P6. 知识按版本隔离，不做跨版本兼容

**✅ 已实施** · 由 P4 驱动

knowledge-server 按 BanvasGL 版本维护独立向量表。引擎升级时创建新表并写入全量种子，旧表保留供回滚。不在同一张表内做版本兼容。

**决策链：** 不同版本的引擎，ViewType/FlowNode/FieldType 可能有增删 -> 旧版本知识对新版本可能产生误导 -> 按版本隔离确保 AI 检索到的永远是当前版本的正确知识。

**约束：**

- 表名格式：knowledge_v{major}.{minor}
- XiangDi 服务请求 knowledge-server 时必须带版本号参数
- 保留最近 3 个版本的旧表，超出自动清理
- 回滚 XiangDi 服务版本时自动切回对应版本的知识表
