# Schema · 架构级决策

> 整体怎么组织——跨越系统边界的数据契约格式与序列化策略。

关系说明：

- engine:A0→A1：A0（engine 域）把「序列化/版本迁移」归为 banvasgl 运行时机制，A1 是该序列化机制在 schema 域的根表达（Full JSON 唯一基座），故 A1 refines engine:A0。
- A1→A2：Full JSON 作为唯一权威数据格式确立后，版本化存储才能明确"每个版本快照存什么"——存的就是 Full JSON 的完整快照。A1 是数据格式决策，A2 是数据存储决策，前者使后者的设计有了明确对象。

## 数据格式

### A1. 全量 JSON 为唯一序列化基座

**✅ 已实施** · refines engine:A0

BanvasGL 的完整序列化格式（Full JSON）是所有数据表示的唯一权威来源。AI Projection 和物料系统都是 Full JSON 的投影（projection），不是独立的格式。

> **A0 机制/策略定位：** A0 把「序列化/版本迁移」明确归为 banvasgl 运行时提供的**机制**。Full JSON 正是该序列化机制对外暴露的唯一权威格式——运行时负责「如何把 Scene 完整序列化/反序列化」这一底层能力，而「派生哪些投影、何时同步」属上层策略。本决策确立的是 schema 域所有格式的机制基座，故回溯 A0。

**决策链：** A0 确立序列化是 banvasgl 运行时机制 → 引擎内部需要完整结构（所有属性、addon 状态、内部 ID）-> AI 需要精简结构（只看语义相关属性）-> 物料需要可复用片段 -> 如果三者各自独立会产生同步问题 -> Full JSON 为运行时序列化机制的单一事实来源，其他格式通过转换函数（toAIProjection / fromAIProjection / toMaterial / fromMaterial）作为投影派生。

**约束：**

- Full JSON 是 Scene 的完整 serialization，包含所有运行时不可推断的状态
- AI Projection 是 Full JSON 的子集投影 + 结构简化（扁平化样式、省略默认值、重命名为语义化 key）
- fromAIProjection() 是 AI 产出进入引擎的唯一入口，负责补全默认值、生成 ID、验证结构合法性
- 物料是 Full JSON 片段 + 元数据（分类、名称、缩略图），序列化/反序列化走相同管线

**反例：**

- AI 直接操作 Full JSON——属性太多、嵌套太深，AI token 浪费严重且容易写错内部字段
- AI Projection 作为独立格式独立存储——两份数据的同步是噩梦，任一方修改后另一方过期

## 版本化存储

### A2. 应用内容版本化三表存储

**✅ 已实施** · 依赖 A1

应用的三份核心数据各自独立版本化：AppContent（页面 JSON）、CollectionSchemaVersion（数据表定义）、CloudFunctionBundle（云函数定义）。采用 append-only 追加写入，每次变更创建新版本记录。

**问题背景：** 拆分前 Application 模型同时承担元数据管理（name、slug、tenantId 等）和内容存储（appJSON——可能几十 KB 的序列化字符串），导致列表查询、权限校验等高频读操作被迫加载巨大的 appJSON 字段。三种内容实体（appJSON、collections、cloudFunctions）的存储方式不一致：有的嵌入 Application 文档原地更新，有的散落多个独立文档无法做原子快照。AI 对话做应用快照时必须全量复制嵌入（因为源表会被后续修改覆盖），每轮对话都冗余存储整份内容。

**核心洞察：** 如果内容实体是 append-only 的版本链，版本回退就退化为"把 Application 的版本指针指回某个历史版本号"——一次字段更新操作，零数据拷贝。由于内容表是 append-only（旧版本永不修改），版本号引用等价于不可变快照——对话和部署的快照从全量嵌入改为三个版本号引用即可冻结历史状态。Application 退化为纯元数据壳 + 版本指针。

**决策链：** 应用数据需要版本管理（回滚、对比、审计）-> 三份数据变更频率和变更粒度不同 -> 绑定在一起会导致"改了一行云函数就创建所有数据的新版本" -> 独立版本化让每份数据按自身节奏演进 -> append-only 保证历史不可篡改。

**约束：**

- 每个版本记录包含：完整快照数据 + 版本号 + 时间戳 + 来源（ai/manual/deploy）
- "当前版本"通过 Application 文档的引用字段指向，修改引用即切换版本
- 历史版本保留策略：最近 50 个版本永久保留，更早的按保留规则清理
- 发布（deploy）时冻结当前三个版本引用为 DeploymentSnapshot
