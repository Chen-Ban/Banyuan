## 产品哲学

### A1. AI 80% 生产 + 人 20% 验收微调

**✅ 已实施**

产品核心模型：AI 负责 80% 的内容生产（页面结构、流程逻辑、数据模型），人负责 20% 的验收、纠偏与微调。用户不写代码，操作的是可视化界面和自然语言。

**行业背景：** AI 生码采纳率已达 90%+，用户在应用生产中的角色发生了根本性转变：从"亲自构建者"变为"意图表达者 + 验收者"。业界主流产品（Lovable、Figma Make、Bolt）正在向同一形态收敛——自然语言生成（粗粒度）+ 可视化微调（细粒度）+ 一键部署。Banyan 的差异化竞争力聚焦在本地化运行 + 桌面安装包分发，而非编辑器功能的完整性。

**决策链：** 目标用户是设计师和业务人员 -> 他们无法（也不应该）写代码 -> AI 生成 + 人审核是唯一可行的交互模型 -> 产品设计围绕"AI 先出草稿、人做微调"展开，而非"人从零搭建"。

**用户控制权底线：** 80/20 哲学的含义是"AI 主导生产"，而不是"剥夺用户的手动能力"。无论 AI 能力多强，凡是会让用户感到失控的地方，都必须保留手动入口——哪怕 99% 的情况下用不到。取舍标准不是"AI 能做就砍掉手动入口"，而是"如果只靠 AI、用户完全没有手动入口，用户会不会失控"。会失控的，保留；不会的，才考虑移除。

**功能取舍原则：**

- 保留：服务于"验收和微调"的功能——直觉化视觉控制、画布直接操作、组件面板（兜底手动入口）、快速预览、一键打包、逻辑视图
- 移除/降优先级：服务于"人主导生产"且移除后用户不会失控的功能——从零手动构建流程图、手动拖拽连线绑定事件、手动数据建模等高认知门槛的编排工具
- 强化：AI 对话入口（AiBar）作为主要生产工具，属性面板作为辅助微调工具
- 评审准则：所有新功能首先问"这是在帮用户生产，还是在帮用户验收/微调？"前者交给 AI，后者才属于编辑器职责

**逻辑视图是验收工具而非生产工具：** AI 生成的交互逻辑（事件触发、数据流动、条件分支）如果完全不可见，业务逻辑对用户而言是黑盒——AI 生成错了用户发现不了，想精确微调也没有入口。因此图形化的逻辑视图属于验收工具应当保留，但其设计原则与传统低代码流程图编辑器有本质区别：只读优先（默认展示 AI 输出的可视化快照），AI 可写（用户框选某段逻辑通过 AiBar 描述修改），人可微调（点击节点修改参数，如 API 地址、条件值），不提供从零手动构建逻辑的能力。判断标准：逻辑视图的每一个交互，都应该回答"这是在帮用户看懂/验收 AI 的输出"，而不是"这是在让用户自己生产逻辑"。

**约束：**

- 所有核心功能必须有 AI 入口（自然语言生成）和手动入口（可视化编辑），不能只有其一
- AI 生成结果必须可回退（对话快照机制），用户可以拒绝 AI 的任何修改
- 设计工具的可视化编辑体验不能因"AI 能做"而降级——20% 的人工微调必须丝滑

**反例：**

- 纯拖拽低代码（无 AI）——搭建效率太低，非技术用户搭复杂应用需要大量时间
- 纯 AI 生成无人工干预——当前 LLM 无法保证 100% 正确，必须有人验收环节
- 完整低代码编辑器（保留所有传统功能，AI 仅作辅助）——大量功能使用频率趋近于零，维护成本高，界面复杂度稀释核心体验

**第一性原理论证：AI Agent 时代为什么还需要可视化平台？**

零代码平台的本质是"意图→可运行软件"的翻译器；AI Agent 的本质也是"意图→可运行软件"的翻译器，只是输入界面从拖拽变成了自然语言。表面上看两者在争夺同一个问题域——如果 Agent 足够聪明，一句话就能得到完整应用，那拖拽式 UI 似乎多余。但这个推理依赖三个隐含假设，逐一质疑如下：

假设一：用户的意图是完整且确定的。现实中大多数用户在开始时并不真正知道自己要什么，他们需要一个"思考的脚手架"——看到半成品才能想清楚下一步。零代码画布提供了这种渐进式探索的空间，纯对话式交互在这点上是弱势的：很难通过连续对话来"看到"一个复杂 UI 的全貌并做微调。

假设二：自然语言是最高效的表达方式。对于"把这个按钮往右移 8px"或"这两个卡片间距再大一点"这类空间性操作，直接拖拽的信息密度远高于语言描述。语言是线性的、离散的；视觉空间操作是并行的、连续的。在精确的视觉微调领域，直接操控仍然是最短路径。

假设三：Agent 的输出是可信且可审计的。当 Agent 生成了 50 个页面的应用，用户怎么验证它是对的？仍然需要一个可视化界面来审查、理解、修改这些产物。这个审查界面本身就是一个"低代码编辑器"。

因此更准确的判断是：纯粹的"拖拽建站"作为唯一交互方式确实在死亡，但零代码平台作为"AI 产物的可视化承载层 + 人类审查/微调界面"不仅不会消失，反而变得更重要。Banyan 的定位正是如此——XiangDi Agent 生成应用骨架，画布提供审查和精调，两者是协作关系而非替代关系。

真正会被淘汰的是两类产品：只有拖拽没有 AI 的零代码平台，以及只有 AI 没有可视化承载的纯对话式 Agent。最终存活的形态是 AI 做 80% 的生成、人在可视化界面里做 20% 的校准和创意决策。Banyan 的价值锚点就在那 20% 的交互体验上——这 20% 必须做到足够好，好到用户愿意留在这里而不是去 IDE 里直接改代码。

## 商业化架构

### A2. 商业化三子系统架构

**未实施** · 依赖 A1

商业化由三个独立子系统组成：支付系统（计费 + 收款）、RBAC 权限控制（角色 + 资源 + 动作）、AI 用量配额（按 token/次数计量 + 配额限制 + 超额策略）。

**决策链：** 平台需要盈利 -> 需要区分免费/付费能力 -> 计费和权限是两个正交维度（付费了不代表有所有权限，有权限不代表不计费）-> 拆为独立子系统各自演进，中间通过"套餐"概念关联。

**约束：**

- 三个子系统通过套餐（Plan）关联：Plan 定义包含哪些权限 + 多少 AI 配额 + 什么价格
- 权限校验是同步的（中间件层拦截），计费是异步的（用量采集 + 定期结算）
- AI 配额超限时降级为排队模式（不直接拒绝），保证用户体验连续性
