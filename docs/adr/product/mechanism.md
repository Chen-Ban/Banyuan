## 计费维度

### M1. AI 用量配额计量机制

**未实施**

AI 用量按两个维度计量：token 消耗（精确计费依据）和调用次数（用户可感知的用量指标）。每次 AI 会话完成后异步上报用量到计量服务，定期汇总写入配额表。

**决策链：** AI 调用有真实成本（LLM API 按 token 计费）-> 需要将成本映射为用户配额 -> token 精确但用户无感，次数粗略但用户可理解 -> 对外展示次数，内部按 token 核算成本。

**约束：**

- 配额检查在请求入口（banyan 后端 AI 代理层），超配额后请求进入排队而非拒绝
- 用量上报是异步的（不阻塞 AI 响应），允许短暂的超额窗口
- 配额重置周期与计费周期对齐（月度）

## 权限维度

### M2. RBAC 权限控制机制

**未实施**

权限模型：角色（Role）绑定一组权限（Permission），用户关联角色。权限粒度为 Resource x Action（如 application:create、ai:chat、deploy:publish）。

**决策链：** 不同套餐用户能使用的功能不同 -> 需要权限控制 -> RBAC 是最成熟的模型 -> 资源+动作的二维权限粒度足以覆盖当前场景。

**约束：**

- 权限校验在 Koa 中间件层统一拦截，业务代码不做权限判断
- 管理员角色拥有所有权限，不走校验逻辑
- 权限定义随功能迭代增量添加，不做一次性全量设计
