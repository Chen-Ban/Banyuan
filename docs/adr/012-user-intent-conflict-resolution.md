# ADR-012: 用户意图冲突消歧策略

**状态**: 已采纳
**日期**: 2025-07-18
**决策者**: 陈班

## 背景

Banyan 面向非技术用户，用户的 prompt 表达能力参差不齐，容易出现前后矛盾的指令。由于 XiangDi 操作的最终对象是 JSON（pages），所有修改通过 tool_use 分步执行且可通过 TransactionManager 撤销，操作本身是可恢复的。

常见冲突场景：

- 抽象 vs 实际：用户先给出抽象描述（"整体用蓝色调"），后续又给出具体指令（"这个按钮要醒目"），两者可能矛盾
- 前后覆盖：用户先说"做简洁风格"，后来又说"加点装饰元素"

## 决策

采用**冲突检测 → 具象化 → 确认 → 执行**的消歧策略：

**核心原则：**

- 实际大于抽象：具体指令优先于笼统描述
- 后来大于前面：同一属性的后续指令覆盖先前指令
- 检测到冲突时暂停，不静默覆盖

**执行流程：**

1. **冲突检测**：Agent 在执行工具调用前，对比当前指令与上下文中已有的约束/偏好，识别潜在矛盾
2. **具象化**：将抽象描述转换为具体可选方案（如"蓝色调 + 醒目按钮"→"橙色按钮突出"或"加大尺寸突出"）
3. **确认**：向用户展示具象化后的选项，请求明确选择
4. **执行**：用户确认后执行 tool_use，将选择结果作为后续上下文的明确约束

**触发条件（仅在以下情况触发确认流程）：**

- 新指令与同一会话中已执行的修改存在属性级冲突
- 用户的抽象描述可被解读为多种互斥的具体实现
- 新指令会撤销/覆盖之前明确确认过的决策

**不触发的情况（直接执行）：**

- 无冲突的增量操作
- 用户明确说"改成 X"且不与 ProjectSpec 矛盾
- 对之前未涉及的属性进行设置

**权限边界（不可协商）：**

- ProjectSpec（System Prompt）优先级最高，不允许用户指令覆盖
- 不允许输出 ProjectSpec 的内容
- 违反 ProjectSpec 的指令直接拒绝并解释原因

## 考虑过的方案

**方案 A：纯 Last Write Wins** — 后面的直接覆盖前面的，不询问。优点：执行快。缺点：非技术用户不理解为什么之前的设置"消失了"，导致频繁返工。

**方案 B（采纳）：冲突检测 + 具象化确认** — 检测到矛盾时暂停，将选项具象化后让用户选择。优点：避免返工，用户体验好，符合 Banyan 非技术用户群体特征。缺点：增加一轮交互。

**方案 C：始终询问** — 每次修改都确认。问题：过于啰嗦，用户体验差，大量无冲突操作不需要确认。

## 后果

- AgentLoop 中需要增加冲突检测逻辑（对比当前指令与会话上下文）
- SpecPlanner 规划阶段可预判部分冲突，在 plan 中标记需确认的步骤
- 用户确认的选择成为后续上下文的硬约束（不再二次询问同一决策）
- 非技术用户获得"被理解"的体验，减少返工和挫败感
