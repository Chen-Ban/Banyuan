# 布局系统后续演进方案

> 临时文件，实施完成后删除

## 当前状态

布局收口改造已完成：
- 脏标记 `_layoutDirty` + `markLayoutDirty()` 向上冒泡
- 统一入口 `renderToOffScreen` 开头触发 layout
- ctx 透传链路：renderToOffScreen → layout → layoutContent → TextFields.layout
- `layout(ctx?): Bounds` 返回实际视口
- `computeScroll()` 独立方法

遗留问题：
- FlexView 未利用子 layout 返回值（子的 layout 由各自 renderToOffScreen 触发，FlexView 只读静态 viewport）
- TextView 同步 layout 仍依赖 OffscreenCanvas fallback（无渲染 ctx 可用）

## 演进方向 1：FlexView 主动布局子节点（递→归完整实现）

### 问题

当前 FlexView.layout() 只读取 `child.viewport` 的静态尺寸来分配空间，但子节点的实际内容可能撑大视口（needStructViewport 模式）。子的 layout 要等到各自 renderToOffScreen 时才执行，此时 FlexView 的布局已经结束，导致：
- 子内容撑大后 FlexView 不知道，布局结果不准确
- 需要两帧才能收敛（第一帧子撑大，第二帧 FlexView 感知到脏标记重排）

### 方案

FlexView.layout() 中主动调用子的 layout，形成真正的"递→归"：

```typescript
public override layout(ctx?: CanvasRenderingContext2D): Bounds {
    this._layoutDirty = false;
    
    // ... flex 空间分配逻辑（递：分配建议区域）...
    
    // 递：主动触发子布局，收集实际尺寸
    for (const child of children) {
        // 将分配的空间写入子的 viewport（建议区域）
        // ...existing flex allocation code...
        
        // 主动执行子布局，获取子的实际视口（归）
        const childActualBounds = child.layout(ctx);
        // 子的 _layoutDirty 已被清除，renderToOffScreen 时不会重复执行
        
        // 用子的实际尺寸（而非建议尺寸）计算主轴游标
        const actualMain = direction === 'row' 
            ? childActualBounds.width 
            : childActualBounds.height;
        cursor += actualMain + gap + extraGap;
    }
    
    // 归：基于子的实际尺寸更新自身 layoutArea
    return super.layout(ctx);
}
```

### 关键设计决策

1. **子 layout 由父主动调用 vs 由 renderToOffScreen 被动触发**
   - 父主动调用：一帧内收敛，但父需要知道子的存在（耦合）
   - 被动触发：解耦，但可能需要多帧收敛
   - **建议**：FlexView 等布局容器主动调用，CombinedView（自由定位）保持被动

2. **避免重复 layout**
   - 父调用 `child.layout(ctx)` 后子的 `_layoutDirty` 已清除
   - 子的 `renderToOffScreen` 检查脏标记时发现已清，跳过 layout
   - 零额外开销

3. **嵌套 FlexView**
   - 外层 FlexView.layout → 内层 FlexView.layout → 内层子 layout
   - 自然递归，一帧内从根到叶全部收敛

### 改动范围

- `FlexView.layout()`：在空间分配后、位置计算中，插入 `child.layout(ctx)` 调用
- 用 `childActualBounds` 替代 `child.viewport` 读取实际尺寸
- 预计 ~30 行改动

## 演进方向 2：消除 OffscreenCanvas fallback

### 问题

TextView 的 `input()`/`delete()`/`newLine()` 同步调用 `this.layout()` 时不在渲染流程中，ctx 为 undefined，TextFields 回退到 OffscreenCanvas 单例测量文字。这个 fallback：
- 在 Node.js 环境下返回 null（后端序列化场景跳过测量）
- 在浏览器中创建额外的 OffscreenCanvas 实例
- 测量结果可能与真实 canvas 有微小差异（字体渲染引擎差异）

### 方案 A：缓存渲染 ctx 引用

```typescript
// View 基类新增
protected _cachedMeasureCtx: CanvasRenderingContext2D | null = null;

// renderToOffScreen 中缓存
private renderToOffScreen(canvasContext: CanvasContext): void {
    const offscreenCtx = canvasContext.getBufferContext();
    this._cachedMeasureCtx = offscreenCtx; // 缓存供非渲染路径使用
    
    if (this._layoutDirty) {
        this.layout(offscreenCtx);
    }
    // ...
}

// layout 中优先使用传入的 ctx，其次使用缓存
public layout(ctx?: CanvasRenderingContext2D): Bounds {
    const measureCtx = ctx ?? this._cachedMeasureCtx;
    // ...
}
```

**优点**：简单，改动小
**缺点**：首次渲染前缓存为 null（但此时 _layoutDirty=true，会走 renderToOffScreen 路径拿到 ctx）
**风险**：canvas 被销毁后缓存失效（需要在 Scene 销毁时清理）

### 方案 B：TextView 编辑操作改为标脏 + 延迟 setSelection

```typescript
public input(content: string, isComposition: boolean): void {
    // ...文本修改逻辑...
    
    // 标脏而非同步布局
    this.markLayoutDirty();
    
    // 延迟设置 selection（等布局完成后）
    this._pendingSelection = { fixed: insertIndex, dynamic: newIndex };
}

// 在 layout 末尾检查并应用 pending selection
public override layout(ctx?: CanvasRenderingContext2D): Bounds {
    const result = super.layout(ctx);
    if (this._pendingSelection) {
        const { fixed, dynamic } = this._pendingSelection;
        this._pendingSelection = null;
        this.setSelection(fixed, dynamic);
    }
    return result;
}
```

**优点**：彻底消除同步 layout，所有路径统一
**缺点**：selection 延迟一帧显示（用户可能感知到闪烁）
**风险**：如果编辑操作连续触发（如长按键盘），pending 会被覆盖

### 建议

**短期采用方案 A**（缓存 ctx），改动最小、风险最低。方案 B 作为长期目标，需要配合输入法 composition 事件的节流机制一起设计。

### 改动范围

方案 A：
- View 基类新增 `_cachedMeasureCtx` 字段
- `renderToOffScreen` 中赋值
- `layout()` 中 fallback 到缓存
- 预计 ~10 行改动

## 演进方向 3：布局缓存与增量更新

### 问题

当前每次 layout 都是全量重算（遍历所有 children、重新 measureChildren）。对于大型场景（100+ View），即使只有一个叶子节点变化，整条祖先链都会重新布局。

### 方案

引入布局缓存，只重算真正变化的子树：

```typescript
// View 基类
private _childrenLayoutCache: Bounds | null = null;

public measureChildren(): Bounds {
    // 如果没有子节点脏，直接返回缓存
    if (this._childrenLayoutCache && !this._hasChildLayoutDirty()) {
        return this._childrenLayoutCache;
    }
    // 重算
    const result = /* ...existing logic... */;
    this._childrenLayoutCache = result;
    return result;
}

private _hasChildLayoutDirty(): boolean {
    return this.children.some(child => child._layoutDirty);
}
```

### 建议

**优先级低**。当前场景规模（通常 < 50 View）下性能不是瓶颈。等出现实际性能问题时再引入，避免过早优化增加复杂度。
